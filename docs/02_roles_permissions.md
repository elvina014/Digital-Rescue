직급별 권한 및 업무 프로세스 명세 (Roles & Permissions)1. 개요본 문서는 login.digital-rescue.com (관리자/직원 포털) 내에서 사용자의 직급(EmployeeRole)에 따른 접근 권한(Access Control)과 데이터 수정 권한(CRUD)을 정의한다.AI(Claude)는 API 라우트와 프론트엔드 컴포넌트(UI)를 구성할 때 이 문서의 제약 조건을 반드시 확인하고, 미권한자의 접근을 차단하는 로직(Middleware / Guard)을 구현해야 한다.2. 직급별 상세 권한 정의2.1 관리자 (ADMIN)모든 시스템 통제권 및 최고 권한을 가진다.접근 가능 메뉴: 전체 대시보드, 통계, 모든 접수건 내역, 재고 관리, 직원 관리, CMS(edit.digital-rescue.com)수행 권한:전사 통계 자료 열람 (지역별 접수율, 채널별 비율, 취소율, 직원별 업무 성취도 등).예외 상황 발생 시, 승인 완료(is_approved = true)된 티켓을 포함한 모든 데이터의 강제 수정 및 삭제.직원 계정 생성 및 권한 부여.2.2 팀장 (MANAGER)운영 전반의 실무 책임자이자 승인권자이다.접근 가능 메뉴: 대시보드, 모든 접수건, 직원별 성과, CMS(edit.digital-rescue.com)수행 권한:모든 접수건 열람 및 수정.웹페이지(대고객 홈페이지) 콘텐츠 수정.센터 승인 권한 (핵심): 담당기사가 올린 자재비 및 최종 견적(final_price) 검토 후 최종 승인 처리 (is_approved를 true로 변경). 최종 승인된 티켓은 관리자, 팀장 외 내용 수정 불가.완료된 건의 최종 승인 및 분쟁 조정.2.3 접수처 (RECEPTION)고객의 첫 대면 채널로, 초기 정보 입력을 담당한다.접근 가능 메뉴: 신규 접수 등록, 신규 접수 목록수행 권한:전화 접수, 내방 접수 등 신규 수리 건(TicketStatus.NEW) 생성.고객 기본 정보 입력 및 고장 증상 기록.생성된 접수건에 대해 담당기사(TECHNICIAN) 배정 권한 (상태를 ASSIGNED로 변경).제한 사항: 본인이 접수한 건이라도, 담당기사가 배정되어 수리가 진행 중인 건의 금액/견적은 수정할 수 없다.2.4 담당기사 (TECHNICIAN)핵심 실무자로, 본인에게 할당된 건만 처리할 수 있다.접근 가능 메뉴: 내 할당 목록(My Tickets), 자재/재고 조회수행 권한:본인에게 배정된 접수건(assignee_id === my_id) 열람 및 수정.기기 사양 체크 및 증빙 자료(사진, 동영상) 업로드.device_models DB를 조회하여 최소 견적 산출 및 최종 견적(final_price) 입력.자재비 입력 후 팀장에게 승인 요청 (상태를 WAITING_APPROVAL로 변경).복잡한 수리의 경우 정밀수리팀(EXPERT_REPAIR)으로 이관 처리.제한 사항: - 타 담당기사의 접수건 열람/수정 불가.팀장 승인 완료(is_approved = true) 후에는 일체 내용 수정 불가.2.5 정밀수리팀 (EXPERT_REPAIR)일반 수리가 불가능하여 이관된 건이나 외주 입고 건을 관리한다.접근 가능 메뉴: 정밀 수리 이관 목록, 자재/재고 조회수행 권한:정밀 수리로 분류된 접수건 열람.수리 진행 상황(로그) 입력 및 자재비 승인 요청.수리 완료 후 상태 변경.2.6 고객 서비스 (CS)수리 완료 후 발생하는 사후 관리(A/S) 및 불만을 처리한다.접근 가능 메뉴: 완료된 접수건 목록(COMPLETED), CS 대기열수행 권한:수리 완료된 내역(견적, 담당기사, 처리 로그) 조회 전용(Read-only).사후 서비스(재수리, 불만 등) 요청 발생 시 CS 로그 입력.필요 시 재처리 지시(상태 업데이트).3. 상태 변경(State Transition) 및 데이터 보호 규칙AI는 API 작성 시 아래의 비즈니스 로직 제약을 무조건 포함해야 한다.상태 보호 (Status Guard): - TicketStatus.COMPLETED 또는 CANCELED 상태의 티켓은 ADMIN 외에는 상태나 내용을 바꿀 수 없다.is_approved === true 일 때, final_price는 절대 업데이트될 수 없다 (서버단 검증 필수).할당 검증 (Assignee Validation): - TECHNICIAN 직급이 티켓을 업데이트(PUT/PATCH) 요청할 때, 서버는 반드시 ticket.assignee_id === current_user.id 인지 확인해야 한다. 일치하지 않으면 403 Forbidden을 반환한다.견적 하한선 강제 (Minimum Price Enforcer):담당기사가 final_price를 입력하여 저장할 때, 백엔드는 해당 모델의 initial_estimate(시스템 최소 견적)과 비교한다.final_price < initial_estimate 인 경우, 즉시 저장을 거부하거나 팀장의 별도 예외 승인 절차를 타도록 강제한다.4. UI 렌더링 지침 (Frontend)메뉴 숨김: 클라이언트(React) 단에서는 사용자 role을 기반으로 접근 권한이 없는 네비게이션 메뉴는 아예 노출하지 않는다.버튼 비활성화: 승인 완료된 폼의 '저장' 버튼은 비활성화(disabled) 처리하고, 입력 필드는 readOnly로 전환하여 사용자에게 직관적인 피드백을 제공한다.
